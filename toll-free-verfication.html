Toll-Free

Required Information for Toll-Free Verification

Toll-Free Verification requires specific data for review. This data identifies the end business sending messages and confirms proper compliance measures. Below you’ll find what’s required, with examples of strong vs. weak submissions.

Note: Refer to the documentation links above for a step-by-step on submitting verification requests via API or Console.

What You Need To Know

The following sections expand on key request parameters from the API docs with plain-language guidance and examples.

Required Data and Examples

Toll-Free phone number SID 🔗

This is Twilio’s phone number SID. If a phone number moves accounts, the SID changes and the existing toll-free verification no longer applies—you’ll need to re-verify.

Business Name 🔗

The end business the customer is engaging with. This should not be the ISV unless:

  • the ISV is the sole content creator,
  • messages are sent on behalf of the ISV, or
  • content is branded with the ISV’s name.

Approved: John’s Coffee Shop

Rejected: Name of the ISV, N/A

Business Website 🔗

Provide the website of the end business. If no traditional site, public social links are acceptable (Facebook, Instagram, etc.).

Approved: Direct URL to end-business website or public social page.

Rejected: Not live, behind login/password, or the ISV/aggregator’s site.

Note: URLs are critical for marketing use cases.

Business Address 🔗

Provide the physical location of the end business.

Approved: 123 Main St, Seattle, WA, 98119 (street, city, state, zip)

Rejected: “N/A”, ISV’s address

Business Contact 🔗

Must reflect the end-user business interacting with the opted-in handset (not the ISV), except where the ISV is the sole content creator and manages opt-in.

Used only for verification; Twilio will not contact the end business.

Approved (example):

  • First Name: Henry
  • Last Name: Saul
  • Email: hsaul@smokepiesbbq.com
  • Phone: +15555555555 (E.164)

Notification Email 🔗

Email to receive verification results.

Approved: notification@twilio.com

Message Volume 🔗

Estimated monthly messages on the toll-free number. Choose the closest value (or where you expect to be in ~6 months).

Options: 10; 100; 1,000; 10,000; 100,000; 250,000; 500,000; 750,000; 1,000,000; 5,000,000; 10,000,000+

Use Case Category 🔗

Select the category that best fits the traffic pattern for this number.

Use Case Summary 🔗

Explain how the business uses messaging on this toll-free number. More detail is better.

Approved: “This number sends promotional offers and coupons to John’s Coffee Shop customers.”

Rejected: “Marketing” (too vague; specify what recipients will receive).

Production Message Sample 🔗

Provide a real-world example of what recipients will receive.

Approved: “Thank you for being a loyal customer of John’s Coffee Shop. Enjoy 10% off your next purchase. Reply STOP to opt out.”

Rejected: “Your appointment is today at 10:00 AM” (doesn’t match a marketing use case).

Opt-In Type 🔗

Opt-in is required (TCPA “express written consent” for marketing). The selected OptInType must match what you demonstrate in your OptInImageUrls.

VERBAL

Example IVR script capturing affirmative consent (with message frequency, rates, HELP/STOP, TOS/Privacy links). Must include script in your submission.

WEB_FORM

Embedded website form with an unchecked checkbox and clear consent language. Provide the public URL.

PAPER_FORM

Physical form collecting phone and consent. Provide a scan/photo.

VIA_TEXT

Keyword campaign (“Text COFFEE to …”). Describe keyword, where it’s advertised, and double opt-in flow.

MOBILE_QR_CODE

QR code to an opt-in form or prefilled message. Provide the QR in your document.

Opt-In Image URLs 🔗

Show exactly how the handset provides consent. Use a public, reachable link to a page, screenshot, photo, or document that clearly demonstrates the opt-in method selected.

  • VERBAL: Include the sample script and flow in a document.
  • WEB_FORM: Link directly to the phone/SMS opt-in page or include a screenshot. Email-only opt-in is not acceptable.
  • PAPER_FORM: Include a scan/photo of the signed form.
  • VIA_TEXT: Document the keyword, where it’s seen, and what happens after texting it (screenshots/URLs help).
  • MOBILE_QR_CODE: Include a document with the QR code.

Approved examples:

  • Direct signup URL (e.g., https://example.com/signup-sms)
  • Public image of the opt-in (e.g., https://example.com/optin.png)
  • Public document describing the opt-in (e.g., https://example.com/optin.pdf)
  • “Keyword: Coffee … reply Y to confirm” with where the keyword appears.

Rejected examples: Homepage URLs that don’t show the opt-in, links behind logins, or vague documents that don’t show where/how consent is given.

Additional Information 🔗

Include anything else that helps reviewers (privacy policy, AUP, onboarding controls, or justification if using multiple numbers for the same business).

Example: https://example.com/privacypolicy
